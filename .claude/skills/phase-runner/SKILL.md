# phase-runner

계획 문서에 정의된 **하나의 phase를 끝까지 완료**시키는 스킬이다. "plan만 쓰기"나 "코드만 짜기"가 아니라, 한 phase를 **계획 → 구현 → 검증 → 기록 → 커밋**의 한 사이클로 닫는 것이 목적이다.

이 스킬이 존재하는 이유: 계획 문서(PLAN-MUST.md 등)는 phase별 작업과 "완료 기준"을 갖고 있지만, 사람이 매번 그 섹션을 읽고 → 하위 작업으로 쪼개고 → 구현하고 → 기준을 하나씩 검증하고 → 문서 체크박스를 갱신하고 → 커밋하는 일은 빠뜨리기 쉽다. 이 스킬은 그 사이클을 일관되게 강제한다.

## 입력

두 가지를 받는다.

1. **문서 경로/링크** — phase들이 정의된 계획 문서. 보통 `docs/PLAN-MUST.md`.
2. **phase 번호** — 예: `1`, `2`.

입력이 빠졌을 때:
- 문서가 없으면 `docs/` 안의 계획 문서(PLAN-*.md 등)를 찾아 후보를 제시하고 확인받는다. 후보가 하나뿐이면 그것으로 진행하되 명시한다.
- phase 번호가 없으면, 문서에서 **완료 기준 체크박스가 아직 비어 있는 가장 앞선 phase**를 "다음 phase"로 제안한다.
- "다음 phase 진행"류 요청은 위 규칙으로 phase를 추론한다.

## 워크플로

아래 6단계를 순서대로 수행한다. 각 단계를 건너뛰지 않는다 — 특히 검증(4)과 체크박스 갱신(5)은 이 스킬의 핵심 가치다.

### 1. 대상 phase 읽기

- 계획 문서를 읽고 지정된 phase 섹션 전체를 파악한다: 목표, 하위 폴더/모듈, 그리고 **"완료 기준"** 목록.
- 그 phase의 **의존 phase가 끝났는지** 확인한다(예: Phase 3는 Phase 1·2 전제). 선행 phase의 완료 기준 체크박스가 비어 있으면 사용자에게 알리고, 그대로 진행할지 확인한다.
- 저장소의 `CLAUDE.md`(루트 및 관련 하위 폴더)를 읽어 아키텍처 규율·디렉토리 역할·코딩 규약을 확보한다. 구현은 반드시 이 규약을 따른다.

### 2. 상세 실행 plan 생성

- 파일 경로: `docs/<theme>-phase-<N>-plan.md`. `<theme>`는 소스 문서에서 딴다 — `PLAN-MUST.md` → `must`, `PLAN-<X>.md` → `<x>`(소문자). 예: `docs/must-phase-1-plan.md`.
- 내용: 해당 phase 섹션을 **구현 가능한 작업 단위(work item)로 분해**한다. 각 작업 항목은 체크박스로, 무엇을·어느 파일에·왜 하는지 한 줄로 적는다. 각 항목에 짧은 ID(예: `W1`, `W2`)를 부여한다.
- **의존성·병렬성 분석** — 작업 항목을 적은 뒤, 항목 간 의존 관계를 분석한다. 각 작업에 대해 "이 작업이 시작되려면 어떤 작업의 산출물이 필요한가"를 따져 **선행 작업(depends-on)** 을 명시한다. 같은 파일을 쓰거나(쓰기 충돌) 한쪽 산출물을 다른 쪽이 import/소비하면 의존 관계다. 서로 다른 파일·모듈을 건드리고 공유 상태가 없으면 병렬 가능하다.
- **병렬 그룹 표기** — 분석 결과를 plan에 **실행 그룹(wave)** 으로 정리한다. 같은 그룹 안의 작업은 서로 의존이 없어 **병렬 실행 가능**하고, 그룹 사이에는 순서 의존이 있다. 아래 형식을 따른다:

  ```markdown
  ## 작업 항목
  - [ ] W1 (geometry) ... — depends-on: 없음
  - [ ] W2 (schema) ...   — depends-on: 없음
  - [ ] W3 (store) ...     — depends-on: W1, W2

  ## 실행 계획 (병렬성)
  - **Wave 1 (병렬 가능):** W1, W2  — 서로 다른 폴더, 공유 상태 없음
  - **Wave 2 (순차):** W3            — W1·W2 산출물에 의존
  ```
- 병렬 그룹이 하나도 없으면(전부 순차) 그 사실과 이유를 한 줄로 명시한다("모든 작업이 선행 산출물에 직렬 의존").
- plan 상단에 소스 문서·phase 번호·생성일을 메타로 적는다. 이 plan 파일이 작업 중 체크리스트 역할을 한다.

문서 작성 톤은 간결하게. 작업 항목은 "추측"이 아니라 소스 phase 섹션과 CLAUDE.md 폴더 역할에 **근거**해야 한다. 병렬성 판단도 마찬가지로 실제 파일·모듈 경계에 근거해야 한다 — 막연히 "독립적으로 보임"이 아니라 "건드리는 파일이 다르고 공유 상태가 없음"처럼 근거를 댄다.

### 3. 구현

- plan의 **실행 계획(병렬성)** 순서대로 구현한다. 같은 wave의 작업은 서로 의존이 없으므로 한 번에(병렬 에이전트 또는 한 응답의 다중 도구 호출로) 진행할 수 있고, 다음 wave는 이전 wave가 끝난 뒤 시작한다. 진행하며 plan 파일의 작업 체크박스를 갱신한다.
- 저장소 규약을 따른다. candle의 경우 특히:
  - **MVVM 레이어 경계**: View(canvas/meshes/panels/ui)는 계산 금지, ViewModel(tools/store 액션/texture)은 렌더 기술(three/r3f) import 금지, Model(schema/geometry/store 상태)은 UI를 모른다.
  - **좌표는 전개도(UV) 기준**, 변환은 `packages/shared/geometry` 순수 함수로.
  - 단일 디자인 문서: 모든 상태는 `document/store`.
- 새 코드에는 검증 가능한 최소 단위 테스트를 함께 둔다(특히 `geometry`·`store`처럼 순수 로직). 이는 4단계 검증을 뒷받침한다.

### 4. 검증 (3종 모두)

세 가지를 **모두** 수행하고 결과를 수집한다.

1. **정적 검사** — 저장소 표준 명령을 실행한다. candle 기준:
   - `pnpm -r typecheck`
   - `pnpm -r build` (또는 영향받은 워크스페이스만)
   - `pnpm -r test`
   - `pnpm lint`
   명령이 다르면 루트 `package.json`의 scripts에서 실제 명령을 확인해 쓴다.
2. **완료 기준 매핑** — 마스터 문서의 해당 phase "완료 기준" 항목을 **하나씩** 짚으며, 그것을 만족함을 보이는 구체 증거(통과한 명령, 작성한 테스트, 실행 결과)를 연결한다. 증거 없이 통과 처리하지 않는다.
3. **런타임 스모크 테스트** — 가능한 경우 실제로 띄워 동작을 확인한다. 예: api는 서버를 잠깐 띄워 헬스/엔드포인트를 curl, web은 빌드 산출물이나 dev 서버로 화면 렌더 확인. 자동 확인이 불가능하면 무엇을 수동 확인해야 하는지 보고에 명시한다.

검증 실패 시: 원인을 고치고 재실행한다. 두 번 이상 같은 곳에서 막히면 멈추고 사용자에게 상황과 옵션을 보고한다(완료 기준을 무리하게 통과 처리하지 말 것).

### 5. 체크박스 갱신

- 검증으로 **증거가 확보된** 완료 기준만 마스터 문서에서 `- [ ]` → `- [x]`로 바꾼다.
- 부분 충족이거나 후속 phase에서 검증할 항목은 체크하지 않고, plan 파일·보고에 사유를 남긴다(예: "3D 반영은 Phase 4에서 검증").
- 7장(있다면)의 교차 검증/완료 정의 중 이번 phase가 실제 달성한 항목이 있으면 함께 체크하고 출처를 주석으로 남긴다.

### 6. 커밋

- `CLAUDE.md`의 git 규약을 따른다. **기본 브랜치(main 등)에 있으면 먼저 작업 브랜치를 만든다** — 단, 사용자가 명시적으로 특정 브랜치/현재 브랜치 커밋을 지시했으면 그에 따른다.
- 변경 전체를 스테이징하고 한국어 커밋 메시지로 커밋한다. 제목은 `feat:`/`chore:` 등 적절한 타입 + phase 요약.
- 커밋 메시지 본문에 구현·검증 요약을 담고, 마지막 줄에 다음을 포함한다:
  ```
  Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
  ```
- push는 하지 않는다(사용자가 요청할 때만).

## 완료 보고 형식

작업을 마치면 다음을 간결히 보고한다.

```
## Phase <N> 완료

**생성한 plan**: docs/<theme>-phase-<N>-plan.md

**구현 요약**: (핵심 변경 2~5줄)

**검증 결과**:
- typecheck/build/test/lint: ✅/❌ (요약)
- 런타임 스모크: (확인한 것 / 수동 확인 필요한 것)

**완료 기준 체크**:
- [x] (기준) — 증거
- [ ] (기준) — 미충족/후속 phase 사유

**커밋**: <hash> <제목> (브랜치: <name>)
```

## 가드레일

- 완료 기준을 **증거 없이 체크하지 않는다.** 이 스킬의 신뢰성은 체크박스가 곧 검증 통과를 의미한다는 데 있다.
- 지정 phase의 범위를 넘어 다른 phase 작업을 끌어오지 않는다. 후속 phase에서 다룰 항목은 명시적으로 미룬다.
- 소스 문서·CLAUDE.md와 충돌하는 추측 구현을 하지 않는다. 모호하면 plan 단계에서 사용자에게 확인한다.
- 검증이 끝까지 통과하지 못하면 커밋하지 않고 사용자에게 보고한다.
