# plan-author

요구사항을 **phase별 실행 계획 문서**로 쓰는 스킬이다. [docs/PLAN-MUST.md](../../../docs/PLAN-MUST.md)가 Must 기능에 대해 하는 일 — 무엇을 어떤 순서로, 어떤 계약·완료 기준으로 만들지 한 문서에 못박는 것 — 을 다른 기능 그룹이나 임의의 요구사항에 대해 동일한 형식으로 재현한다.

이 스킬이 존재하는 이유: PRD나 요구사항은 "무엇(what)"을 말하지만 "어떻게·어떤 순서·어떤 완료 기준(how/order/contract)"은 비어 있다. 사람이 매번 기능을 phase로 쪼개고, 의존 순서를 따지고, 각 phase의 산출물과 검증 가능한 완료 기준을 적는 일은 형식이 들쭉날쭉해지기 쉽다. 이 스킬은 그 산출물을 **일관된 골격**으로 만들어, 곧바로 `phase-runner` 스킬이 한 phase씩 집어 실행할 수 있게 한다.

## 역할 경계 (phase-runner와의 관계)

- **plan-author (이 스킬)**: 계획 문서를 *쓴다*. 코드를 구현하지 않는다. 산출물은 `docs/PLAN-<GROUP>.md` 하나.
- **phase-runner (별도 스킬)**: 이 문서의 *한 phase를 실행*한다(상세 plan → 구현 → 검증 → 체크박스 갱신 → 커밋).

따라서 이 스킬의 출력은 **phase-runner가 소비 가능한 형식**이어야 한다. 구체적으로:
- 파일명은 `PLAN-<GROUP>.md` (phase-runner가 `<theme>`를 `PLAN-<X>.md` → `<x>`로 딴다).
- 각 phase는 `## Phase N — ...` 헤더로 시작해 위치를 찾을 수 있어야 한다.
- 완료 기준은 `- [ ]` 체크박스로 적어, phase-runner가 증거 확보 후 `- [x]`로 바꿀 수 있어야 한다.
- 의존 그래프, 교차 검증(7장), 범위 밖(8장) 섹션을 둔다 — phase-runner가 의존 phase·교차 검증을 참조한다.

## 입력

1. **무엇을 계획할지** — 둘 중 하나:
   - **PRD 기능 그룹**: `CLAUDE.md`의 PRD에서 그룹(예: Should = PRD-S1~S5, Could = PRD-C1~C4). 이 경우 **CLAUDE.md PRD가 단일 출처(SSOT)** 다. 기능 ID·수용 기준·폴더 역할을 거기서 가져오고 재정의하지 않는다.
   - **임의 요구사항 텍스트**: PRD에 없는 자유 형식 목표·기능 설명. 이 경우 요구사항을 먼저 정리(목표·범위·제약)한 뒤 phase로 분해한다.
2. **출력 경로/그룹 이름**(선택) — 기본은 `docs/PLAN-<GROUP>.md`.

입력이 빠졌을 때:
- 무엇을 계획할지 모호하면 `CLAUDE.md`를 읽어 가능한 기능 그룹(Should/Could 등)을 후보로 제시하고 확인받는다.
- 그룹 이름이 없으면 내용에서 합리적으로 정한다(예: Should → `PLAN-SHOULD.md`). 자유 요구사항이면 짧은 주제어로 짓고 사용자에게 확인한다.

## 워크플로

### 1. 소스 파악

- **항상 먼저 `CLAUDE.md`(루트 및 관련 하위 폴더)를 읽는다.** 아키텍처 원칙·MVVM 규율·디렉토리 역할·구현 순서·리스크가 계획의 근거다. PRD 그룹이면 해당 기능 ID와 수용 기준을 그대로 인용한다.
- 기존 `docs/PLAN-*.md`가 있으면 형식·톤·번호 체계를 맞추기 위해 참고한다(특히 PLAN-MUST.md). 이미 만든 phase 번호 체계와 충돌하지 않게 한다.
- 자유 요구사항이면 입력에서 **목표 / 범위 / 제약 / 비범위**를 먼저 1~2문단으로 정리한다.

### 2. phase 분해

- 기능/요구사항을 **의존 순서**로 phase에 배치한다. 토대(데이터·스키마·좌표·도메인 규칙)가 먼저, 그 위에 UI·통합이 온다. CLAUDE.md "구현 순서"가 있으면 그것을 우선 따른다.
- **리스크 선검증**: 가장 불확실하거나 막히면 전체가 흔들리는 부분을 앞 phase에서 PoC로 검증하도록 배치한다(예: PLAN-MUST의 하트 전개도↔UV). 막혔을 때의 대안(범위 축소·후순위 이관)도 한 줄로 남긴다.
- 각 phase의 **의존 phase**와 **병렬 가능 여부**를 정한다. 서로 다른 폴더·모듈을 건드리고 공유 상태가 없으면 병렬 가능. 이 판단은 6장 의존 그래프에 반영한다.
- phase 수는 기능 양에 맞춘다 — 억지로 늘리거나 한 phase에 몰지 않는다. 한 phase는 "한 사람이 한 흐름으로 끝낼 수 있는 묶음"이 기준.

### 3. 각 phase 작성

phase마다 아래 3요소를 반드시 채운다.

1. **목표** — 한 줄. 이 phase가 끝나면 무엇이 가능해지는지.
2. **작업/산출물 테이블** — `| 작업 | 산출물 |`. 작업은 동사로, 산출물은 *구체적 파일·모듈·함수*로 적는다("이것저것 구현"이 아니라 "`packages/shared/geometry`에 `getNet(shape, spec)` 추가"). 필요하면 하위 절(`### N.1`, `### N.2`)에서 **계약·인터페이스·함수 시그니처**를 명시한다 — phase-runner가 추측 없이 구현할 수 있도록.
3. **완료 기준** — `- [ ]` 체크박스 목록. **객관적으로 검증 가능**해야 한다. PRD 그룹이면 해당 기능 ID의 **수용 기준을 그대로 매핑**하고 `(PRD-Sx 수용 기준)`처럼 출처를 단다. "잘 동작한다" 같은 모호한 문장 금지 — "X 명령이 통과한다 / Y 입력에 Z가 반영된다 / 단위 테스트 통과"처럼 증거를 댈 수 있게 쓴다.

phase 사이 의존이 있으면 phase 머리에 `**의존**: Phase N.` 을 적는다.

### 4. 횡단 섹션 작성

문서 전체에 다음 섹션을 둔다(번호·제목은 PLAN-MUST.md를 따른다).

- **0. 현재 상태와 전제** — 레포 현황, 스택 가정(PRD/CLAUDE.md에서 도출), 시작 지점.
- **1. 아키텍처 불변식** — 모든 phase에서 강제할 규칙(CLAUDE.md의 MVVM·좌표 단일화·레이어 경계 등). "PR 리뷰 체크리스트로도 쓴다"고 명시.
- **6. 의존 그래프 / 권장 순서** — phase 간 의존을 ASCII 트리로. 병렬 가능 지점·리스크 PoC 분기를 주석으로.
- **7. 교차 검증 (완료 정의)** — 개별 phase를 넘는 횡단 검증 항목 체크박스(예: 동기화 회귀, 좌표 단일화 grep, 공유 왕복). phase-runner가 phase 완료 시 해당 항목도 체크한다.
- **8. 범위 밖 (혼동 방지)** — 이 계획에서 다루지 *않는* 것과, 그럼에도 확장을 막지 않도록 남길 타입·인터페이스 훅.

### 5. 검토

문서를 다시 읽고 확인한다:
- 모든 PRD 기능 ID(해당 그룹의)가 정확히 한 phase에 배정되었는가? 빠지거나 중복은 없는가?
- 모든 완료 기준이 증거로 검증 가능한 문장인가? PRD 수용 기준과 매핑되는가?
- phase 헤더·체크박스·파일명이 phase-runner 호환 형식인가?
- 작업/산출물이 실제 폴더 역할(CLAUDE.md)과 일치하는가? 레이어 경계를 위반하는 배치는 없는가?

## 문서 골격

```markdown

> 기능 ID·아키텍처·폴더 역할의 정의는 항상 루트 CLAUDE.md를 따른다. 본 문서는 *어떻게(how)* 와 *순서·계약·완료 기준* 을 다룬다.
```

## 작성 원칙

- **what과 how를 분리한다.** 기능 ID·수용 기준·폴더 역할 정의는 CLAUDE.md(SSOT)에 있다 — 재정의하지 말고 인용·참조한다. 이 문서는 순서·계약·완료 기준만 더한다.
- **완료 기준은 곧 검증 계약이다.** phase-runner가 이 체크박스 하나하나에 증거를 붙여 통과시킨다. 그러므로 증거를 댈 수 없는 기준은 쓰지 않는다. 수용 기준이 있으면 그대로 매핑하는 것이 가장 안전하다.
- **산출물은 구체적으로.** "어디에 무엇을"이 분명해야 phase-runner가 추측 없이 구현한다. 가능하면 함수 시그니처·라우트·타입까지 적는다.
- **의존과 병렬을 실제 경계로 판단한다.** "독립적으로 보임"이 아니라 "건드리는 파일이 다르고 공유 상태가 없음"처럼 근거를 댄다.
- **확장 훅을 남긴다.** 지금 범위 밖이라도 나중 기능을 막지 않도록 타입·인터페이스에 자리를 남기라고 8장에 적는다(예: `Element.type`에 미래 값, 선언만 하는 타입).
- 톤은 PLAN-MUST.md에 맞춰 간결하게. 한국어로 쓴다.

## 가드레일

- **코드를 구현하지 않는다.** 이 스킬은 문서만 만든다. 사용자가 구현까지 원하면 문서 작성 후 phase-runner를 안내한다.
- **CLAUDE.md와 충돌하는 추측을 문서에 넣지 않는다.** 폴더 역할·아키텍처와 어긋나는 배치가 필요하면 멈추고 사용자에게 확인한다.
- **기능을 빠뜨리거나 중복 배정하지 않는다.** 대상 그룹의 모든 기능 ID가 정확히 한 phase에 들어가야 한다.
- 완료 기준을 모호하게 쓰지 않는다 — 검증 불가능한 기준은 phase-runner를 무력화한다.
